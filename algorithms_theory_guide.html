Notes

When computing lcm(a,b) be careful with overflow: compute as a / gcd(a,b) * b to divide first. For multiple numbers, lcm(a,b,c) = lcm(lcm(a,b), c). The lcm is useful for periodicity problems, synchronizing cycles, and solving linear diophantine congruences.

Modular Arithmetic

The mathematics of remainders — the foundation of competitive programming

Modular arithmetic defines arithmetic operations within a fixed range [0, m−1]. The expression a mod m gives the remainder when a is divided by m.

a mod m = a − m·⌊a/m⌋
Result:

Core Properties

Addition

(a + b) mod m = ((a mod m) + (b mod m)) mod m

Example: (17 + 8) mod 5 = 25 mod 5 = 0
Same as: (2 + 3) mod 5 = 0 ✓

Subtraction

(a − b) mod m = ((a mod m) − (b mod m) + m) mod m

Add m to avoid negative results

Multiplication

(a × b) mod m = ((a mod m) × (b mod m)) mod m

Example: (7 × 8) mod 5 = 56 mod 5 = 1
Same as: (2 × 3) mod 5 = 1 ✓

Division (not straightforward)

(a / b) mod m ≠ (a mod m) / (b mod m)

Use Modular Inverse instead!
(a / b) mod m = (a × b⁻¹) mod m

Theory: Modular arithmetic operates in Z_m; reduce intermediates to keep values bounded. Division requires a modular inverse (exists iff gcd(b,m)=1); use ((a % m)+m)%m for the canonical representative in [0,m−1].

Visual: Clock Arithmetic (mod 12)

Modular Arithmetic mod 10⁹+7

In competitive programming, results are typically taken mod 10⁹+7 (a prime) to avoid overflow and enable modular inverse via Fermat's little theorem. 

const MOD = 1e9 + 7;
long long add(ll a, ll b) { return (a + b) % MOD; }
long long mul(ll a, ll b) { return (a * b) % MOD; }

Key Properties Summary

Property Formula Example (mod 7)
Congruence a ≡ b (mod m) if m|(a−b) 16 ≡ 2 (mod 7)
Reflexive a ≡ a (mod m) 5 ≡ 5 (mod 7)
Symmetric a ≡ b → b ≡ a (mod m) 9 ≡ 2 → 2 ≡ 9
Transitive a ≡ b, b ≡ c → a ≡ c 16≡2, 9≡2 → 16≡9
Fermat's Little a^(p-1) ≡ 1 (mod p), p prime 3^6 ≡ 1 (mod 7)

Further Theory

Modular arithmetic forms the ring Z_m. When m is prime, Z_m is a field: every non-zero element has a multiplicative inverse. Concepts like units (elements with inverses), zero-divisors (when m is composite), and Euler's totient function φ(m) (count of units) are central. Use Euler/Fermat theorems and the Chinese Remainder Theorem to reduce problems into smaller moduli.

Binary Exponentiation

Computing a^n in O(log n) instead of O(n)

Also called fast power or exponentiation by squaring. Reduces computing a^n from n multiplications to just log₂(n) multiplications.

Key Idea: Express exponent in binary. a^13 = a^(1101₂) = a^8 × a^4 × a^1

Theory: Binary exponentiation reduces exponentiation to O(log n) multiplications by repeated squaring. In modular arithmetic use reductions after multiplications to keep intermediates small; the algorithm is branch-free when implemented via fixed-bit loops. 



        
The Recurrence

        

          a^n = { 1 if n=0 | a^(n/2) × a^(n/2) if n is even | a × a^(n-1) if n
          is odd }
        


        
Trace: 3^13

        

          
        


        
Proof & Complexity

        

          Binary exponentiation follows from writing the exponent in base 2 and
          repeatedly squaring the base. Each loop iteration halves the exponent,
          so the number of multiplications is at most 2⌊log₂ n⌋ (squares +
          occasional multiplies), giving O(log n) time. The algorithm is correct
          by induction on the exponent: splitting n into even/odd cases
          preserves equality.
        


        
Implementations

        

          

            
Iterative (Preferred)

            
long long power(ll a, ll b, ll mod) {
  ll res = 1;
  a %= mod;
  while (b > 0) {
    if (b & 1)        // bit is set
      res = res * a % mod;
    a = a * a % mod; // square base
    b >>= 1;         // shift right
  }
  return res;
}

          

          

            
Recursive

            
long long power(ll a, ll b, ll mod) {
  if (b == 0) return 1;
  ll half = power(a, b/2, mod);
  ll res = half * half % mod;
  if (b % 2 == 1)
    res = res * a % mod;
  return res;
}

          

          

            Theory: This guide groups techniques by problem patterns. Learn the
            invariants each technique maintains (monotonicity, associativity,
            unimodality) and map problems to the smallest tool that preserves
            those invariants — this keeps solutions simple and efficient.

Euclidean Algorithm

        

          Computing Greatest Common Divisor in O(log min(a,b))
        

        

          gcd(a, b) = gcd(b, a mod b), with base case gcd(a, 0) = a
        

        

          Based on the fact that if d divides both a and b, it also divides (a
          mod b).
        

        
Correctness & Complexity

        

          The Euclidean algorithm is correct because gcd(a,b)=gcd(b,a mod b) —
          any common divisor of a and b divides a−q·b, and conversely. The
          algorithm terminates since remainders strictly decrease and are
          non-negative. Worst-case input sizes follow consecutive Fibonacci
          numbers, giving Θ(log_φ(min(a,b))) steps where φ≈1.618, so the
          algorithm runs in O(log min(a,b)) time.
        


        
Trace: gcd(48, 18)

        

          
        


        
Implementations

        

          

            
Iterative

            
int gcd(int a, int b) {
  while (b != 0) {
    int t = b;
    b = a % b;
    a = t;
  }
  return a;
}

          

          

            
Recursive

            
int gcd(int a, int b) {
  if (b == 0) return a;
  return gcd(b, a % b);
}
// C++17: __gcd(a, b) or std::gcd

          

        


        
Extended Euclidean Algorithm

        

          Finds x, y such that ax + by = gcd(a, b). Used to compute
          modular inverse.
        

        
int extgcd(int a, int b, int& x, int& y) {
  if (b == 0) { x = 1; y = 0; return a; }
  int x1, y1;
  int g = extgcd(b, a % b, x1, y1);
  x = y1;
  y = x1 - (a / b) * y1;
  return g;
}

        

          O(log min(a,b)) —
          Fibonacci numbers are the worst case, giving exactly log_φ(min(a,b))
          steps.
        

      


      
      

        
Number Theory

        
GCD vs LCM

        
Greatest Common Divisor vs Least Common Multiple

        

          

            
GCD — Greatest Common Divisor

            

              Definition: Largest number that divides both a
              and b
            

            
Example: gcd(12, 18) = 6

            

              Divisors of 12: {1,2,3,4,6,12}
Divisors of
              18: {1,2,3,6,9,18}
Largest common:
              6
            

          

          

            
LCM — Least Common Multiple

            

              Definition: Smallest number divisible by both a
              and b
            

            
Example: lcm(12, 18) = 36

            

              Multiples of 12: {12,24,36,48,...}
Multiples
              of 18: {18,36,54,...}
Smallest common:
              36
            

          

        


        
The Key Relationship

        
lcm(a, b) = (a × b) / gcd(a, b)

        

          

            Why? Using prime factorization: gcd takes the
            minimum of each prime's exponent, lcm takes the
            maximum.
          

          
12 = 2² × 3¹    18 = 2¹ × 3²

          
gcd = 2^min(2,1) × 3^min(1,2) = 2¹ × 3¹ = 6

          
lcm = 2^max(2,1) × 3^max(1,2) = 2² × 3² = 36

        

        

          Theory: GCD and LCM are dual via prime exponents: gcd uses minima, lcm
          uses maxima. The identity a×b = gcd(a,b)×lcm(a,b) follows from this
          factor-wise decomposition and is useful for reducing overflow by
          dividing before multiplying.
        


        
Venn Diagram of Prime Factors

        

          
        


        
Optimizations & Variants

        

          Practical optimizations include starting marking at p² (because
          smaller multiples were marked by smaller primes), iterating only over
          odd numbers to halve memory and time, and using a segmented sieve to
          generate primes in ranges [L,R] when n is large (memory-limited).
          Wheel factorization (skipping multiples of small primes like 2,3,5)
          further reduces constant factors. For extremely large limits, use a
          bitset and segmented marking for cache efficiency.
        


        
Implementation

        
long long lcm(ll a, ll b) {
  return a / gcd(a, b) * b; // divide first to avoid overflow
}

Number Theory

        
Modular Inverse

        
The modular equivalent of division

        
The modular inverse of a modulo m is a number x such that:

        
a × x ≡ 1 (mod m)

        

          Written as x = a⁻¹ (mod m). Exists if and only if gcd(a, m) = 1 (a and
          m are coprime).
        


        
Existence & Theory

        

          An inverse exists exactly for units of Z_m (elements coprime to m).
          For prime modulus p every non-zero element is a unit and inverses can
          be computed via Fermat's little theorem: a^{p-2} (mod p). For
          composite m, use the extended Euclidean algorithm which returns Bézout
          coefficients x,y with ax+my=g where g=gcd(a,m); if g=1 then x is the
          inverse modulo m. Euler's theorem generalizes Fermat: a^{φ(m)} ≡ 1
          (mod m) when gcd(a,m)=1, so a^{φ(m)-1} is an inverse (needs φ(m)).
        


        

          
        


        

          

            
Method 1: Fermat's Little Theorem

            
When m is prime: a⁻¹ ≡ a^(m-2) (mod m)

            
ll modinv(ll a, ll m) {
  return power(a, m-2, m);
}
// Works when m is prime (e.g. 1e9+7)

            O(log m)
          

          

            
Method 2: Extended Euclidean

            
Works even when m is not prime (but gcd(a,m)=1)

            
ll modinv(ll a, ll m) {
  ll x, y;
  int g = extgcd(a, m, x, y);
  if (g != 1) return -1; // no inverse
  return (x % m + m) % m;
}

            O(log m)
          

        


        
Precomputing All Inverses (1 to n)

        
inv[1] = 1;
for (int i = 2; i <= n; i++)
  inv[i] = (MOD - (MOD/i) * inv[MOD%i] % MOD) % MOD;
// O(n) — much faster than computing each separately


        
Applications

        

          

            
Division in Modular Arithmetic

            
Compute (a/b) mod m safely:
(a × b⁻¹) mod m

          

          

            
Binomial Coefficients

            
C(n,r) mod p = n! × (r!)⁻¹ × ((n-r)!)⁻¹ mod p

          

          

            
Fraction Representation

            
Represent p/q as p × q⁻¹ mod m to avoid fractions entirely

          

        


        
Example

        

          
Find the modular inverse of 3 mod 7:

          
We need x such that 3x ≡ 1 (mod 7)

          

            Try: 3×1=3, 3×2=6, 3×3=9≡2, 3×4=12≡5, 3×5=15≡1 ✓
          

          

            So 3⁻¹ ≡ 5 (mod 7). Using Fermat: 3^(7-2) = 3^5 = 243 = 34×7+5 ≡
            5 ✓

Sieve of Eratosthenes

        
Find all primes up to n in O(n log log n)

        

          Ancient Greek algorithm: mark all multiples of each prime as
          composite, starting from 2.
        


        

          Theory: The sieve marks composites by crossing out multiples of each
          prime; its complexity is O(n log log n) due to harmonic series of
          reciprocals of primes. Use wheel factorization or segmented sieve for
          memory-friendly ranges and for very large n.
        


        
Step-by-Step Visualization (n=30)

        

          
          
        

        

          
        


        
Implementation

        
vector<bool> sieve(int n) {
  vector<bool> is_prime(n + 1, true);
  is_prime[0] = is_prime[1] = false;
  for (int p = 2; (ll)p*p <= n; p++) {
    if (is_prime[p]) {
      for (int j = p*p; j <= n; j += p)
        is_prime[j] = false; // mark multiples
    }
  }
  return is_prime;
}

Combinatorics

        
Binomial Coefficient C(n, r)

        
Number of ways to choose r items from n items

        
C(n, r) = n! / (r! × (n−r)!)

        

          Also written as ⁿCᵣ, C(n,r), or
          ( n ) — "n choose r"
        


        

          Theory: C(n,r) counts unordered selections and satisfies symmetry
          C(n,r)=C(n,n−r) and the recurrence C(n,r)=C(n−1,r−1)+C(n−1,r). Use
          factorials for exact values and precomputed factorial+inverse for many
          queries; complexity is O(1) per query after O(n) preprocessing.
        


        

          
Example: C(5,2)

          
Choose 2 from {A, B, C, D, E}:

          

            AB, AC, AD, AE, BC, BD, BE, CD, CE, DE = 10 pairs
          

          
C(5,2) = 5! / (2! × 3!) = 120 / (2 × 6) = 10 ✓

        


        

          
        


        
Key Properties

        

          

            
Symmetry

            C(n, r) = C(n, n-r)
            

              Choosing r items is same as leaving out n-r items
            

          

          

            
Pascal's Identity

            C(n, r) = C(n-1, r-1) + C(n-1, r)
            

              Either include item n (C(n-1,r-1)) or exclude it (C(n-1,r))
            

          

          

            
Boundary

            C(n, 0) = C(n, n) = 1
            

              Choose none or all — only 1 way each
            

          

          

            
Sum of Row

            Σ C(n, k) for k=0..n = 2ⁿ
            

              Total subsets of n items
            

          

        


        
Computation (Large n, mod p)

        
// Precompute factorials and inverse factorials
const int MOD = 1e9+7, MAXN = 1e6+5;
ll fact[MAXN], inv_fact[MAXN];

void precompute() {
  fact[0] = 1;
  for (int i = 1; i < MAXN; i++)
    fact[i] = fact[i-1] * i % MOD;
  inv_fact[MAXN-1] = power(fact[MAXN-1], MOD-2, MOD);
  for (int i = MAXN-2; i >= 0; i--)
    inv_fact[i] = inv_fact[i+1] * (i+1) % MOD;
}

ll nCr(int n, int r) {
  if (r < 0 || r > n) return 0;
  return fact[n] * inv_fact[r] % MOD * inv_fact[n-r] % MOD;
}

        O(1) per query after
        O(n) precomputation
      


      
      

        
Combinatorics

        
Pascal's Triangle

        
The triangular array of binomial coefficients

        

          Each entry is the sum of the two entries above it. Row n contains
          C(n,0), C(n,1), ..., C(n,n).
        


        

          Theory: Pascal's triangle encodes binomial coefficients and reveals
          identities (binomial theorem, hockey-stick identity). Row sums are 2ⁿ
          and diagonal sums relate to combinatorial sequences (Fibonacci, etc.).
          It is a useful tabulation for combinatorics and DP.
        


        

          
        


        
Building Pascal's Triangle

        
int C[MAXN][MAXN];
void buildPascal(int n) {
  for (int i = 0; i <= n; i++) {
    C[i][0] = C[i][i] = 1;
    for (int j = 1; j < i; j++)
      C[i][j] = (C[i-1][j-1] + C[i-1][j]) % MOD;
  }
}

        O(n²) time and space

        
Hidden Patterns in Pascal's Triangle

        

          

            
Fibonacci

            

              Sum of shallow diagonals gives Fibonacci numbers: 1, 1, 2, 3, 5,
              8...
            

          

          

            
Powers of 2

            
Row sum = 2ⁿ; substituting x=1 in (1+x)ⁿ

          

          

            
Sierpinski

            

              Color odd entries: produces Sierpinski triangle fractal pattern
            

          

        

      


      
      

        
Combinatorics

        
Catalan Numbers

        
A remarkably universal counting sequence

        
Cₙ = C(2n, n) / (n+1) = (2n)! / ((n+1)! × n!)

        
Sequence: C₀=1, C₁=1, C₂=2, C₃=5, C₄=14, C₅=42, C₆=132...


        

          Recurrence: Cₙ = Σ Cᵢ × Cₙ₋₁₋ᵢ for i=0..n-1, with C₀=1
        


        

          Theory: Catalan numbers count many recursive structures
          (parenthesizations, binary trees, noncrossing partitions). Growth is
          ~4ⁿ/(n^{3/2}√π). Use the recurrence for DP (O(n²)) or direct formula
          via binomial coefficients with modular inverse for large n.

1. Valid Parentheses

            

              Cₙ = number of valid bracket sequences with n pairs of parentheses
            

            

              n=3: ()()() ()(()) (())() ((())) (()()) → 5
            

          

          

            
2. Binary Search Trees

            

              Cₙ = number of structurally unique BSTs with n nodes (keys 1..n)
            

            
For n=3: 5 distinct BST shapes exist

          

          

            
3. Non-crossing Handshakes

            

              2n people on a circle, Cₙ ways to pair them with non-crossing
              lines
            

          

          

            
4. Monotonic Lattice Paths

            

              Cₙ = paths from (0,0) to (n,n) that never cross above the diagonal

Techniques

        
Two Pointer Technique

        

          O(n) solutions to problems typically requiring O(n²)
        

        

          Maintain two indices that move through an array — often toward each
          other, or both in the same direction. Eliminates redundant
          recomputation by exploiting monotonicity.
        


        

          Theory: Two-pointer transforms O(n²) checks into O(n) by ensuring that
          each pointer moves monotonically at most n steps. Correctness relies
          on monotonicity or sorted order; complexity is O(n) and space O(1).
        


        
Pattern 1: Opposite Ends (Two Sum in sorted array)

        

          
        

        
// Two Sum in sorted array
bool twoSum(vector<int>& arr, int target) {
  int l = 0, r = arr.size() - 1;
  while (l < r) {
    int sum = arr[l] + arr[r];
    if (sum == target) return true;
    else if (sum < target) l++;
    else r--;
  }
  return false;
}


        
Pattern 2: Fast-Slow Pointers

        
// Count pairs with sum < k (unsorted? sort first)
sort(arr.begin(), arr.end());
int l = 0, r = n-1, count = 0;
while (l < r) {
  if (arr[l] + arr[r] < k) {
    count += r - l; // all pairs (l, l+1..r) are valid
    l++;
  } else r--;
}

Techniques

        
Sliding Window Algorithm

        

          Efficiently process subarrays / substrings using a moving window
        

        

          Instead of recomputing from scratch for each subarray, maintain a
          running aggregate as the window slides — adding the new element and
          removing the old one.
        


        

          Theory: Sliding-window reduces repeated work by updating aggregates
          incrementally. Fixed-window problems cost O(n); variable-window
          problems also run in O(n) when pointers only advance forward. Maintain
          window invariants carefully for correctness.
        


        
Fixed Window: Max sum subarray of size k

        

          
        


        
// Fixed window: max sum of size k
int maxSum(vector<int>& a, int k) {
  int sum = 0;
  for (int i = 0; i < k; i++) sum += a[i]; // initial window
  int mx = sum;
  for (int i = k; i < a.size(); i++) {
    sum += a[i] - a[i-k]; // slide window
    mx = max(mx, sum);
  }
  return mx;
}


        
Variable Window: Longest subarray with sum ≤ k

        
int longestSubarray(vector<int>& a, int k) {
  int l = 0, sum = 0, ans = 0;
  for (int r = 0; r < a.size(); r++) {
    sum += a[r];           // expand right
    while (sum > k)
      sum -= a[l++];       // shrink left
    ans = max(ans, r - l + 1);
  }
  return ans;
}

Techniques

        
Greedy Algorithms

        
Make the locally optimal choice at each step

        

          A greedy algorithm builds up a solution piece by piece, always
          choosing the next piece that offers the most obvious and immediate
          benefit — without reconsidering past choices.
        


        

          Theory: Greedy algorithms are fast (often O(n log n) or O(n)) but
          require a proof (exchange/induction) that local choices lead to a
          global optimum. Recognize safe greedy problems by identifying the
          greedy-choice property and optimal substructure.
        


        
Characteristics

        

          

            
Greedy Choice Property

            

              A globally optimal solution can be reached by making locally
              optimal (greedy) choices. The choice made at each step must be
              safe — it must lead to some optimal solution.
            

          

          

            
Optimal Substructure

            

              The optimal solution to the problem contains optimal solutions to
              its subproblems. (Same as DP, but greedy also needs the greedy
              choice property.)
            

          

        


        
Classic Example: Activity Selection

        

          
        


        
// Sort by finish time, always pick earliest finish
sort(activities.begin(), activities.end(),
  [](auto& a, auto& b) { return a.finish < b.finish; });

int lastFinish = -1, count = 0;
for (auto& act : activities) {
  if (act.start >= lastFinish) {
    count++;
    lastFinish = act.finish;
  }
}

Segment Tree

        
Range queries and point updates in O(log n)

        

          A segment tree is a binary tree where each node stores aggregate
          information (sum, min, max, etc.) about a segment (subarray) of the
          input array. The root covers the entire array; leaves represent
          individual elements.
        


        

          Key insight: Any range [l, r] can be decomposed into O(log n) segments
          stored in the tree.
        


        

          Theory: Segment trees support associative merges; queries and point
          updates touch O(log n) nodes because the interval splits along tree
          levels. Build is O(n) and space O(n) (commonly 4n). They are flexible
          for many range operations.
        


        
Structure for array [2, 4, 5, 7, 2, 3, 6, 1]

        

          
        


        

          

            Operation
            Complexity
            Description
          

          

            Build
            O(n)
            Bottom-up fill all nodes
          

          

            Point Update
            O(log n)
            Update leaf, propagate up
          

          

            Range Query
            O(log n)
            Merge O(log n) segments
          

          

            Space
            O(n)
            4n array is safe
          

        

      


      
      

        
Data Structures

        
Segment Tree Construction

        
Building, querying, and updating

        
Implementation


        

          Theory: Construction initializes node aggregates bottom-up in O(n).
          Choose recursive or iterative (segment tree on array) builds depending
          on memory and constant-factor needs; iterative variants often have
          better cache performance.
        


        
Implementation

        
const int MAXN = 1e5 + 5;
int tree[4*MAXN], arr[MAXN];

// Build: O(n)
void build(int node, int start, int end) {
  if (start == end) {
    tree[node] = arr[start]; // leaf
    return;
  }
  int mid = (start + end) / 2;
  build(2*node,   start, mid);   // left child
  build(2*node+1, mid+1, end); // right child
  tree[node] = tree[2*node] + tree[2*node+1];
}

// Range sum query: O(log n)
int query(int node, int start, int end, int l, int r) {
  if (r < start || end < l) return 0; // no overlap
  if (l <= start && end <= r) return tree[node]; // total overlap
  int mid = (start + end) / 2;
  int left  = query(2*node,   start, mid,   l, r);
  int right = query(2*node+1, mid+1, end, l, r);
  return left + right;
}

// Point update: O(log n)
void update(int node, int start, int end, int idx, int val) {
  if (start == end) {
    arr[idx] = tree[node] = val; return;
  }
  int mid = (start + end) / 2;
  if (idx <= mid) update(2*node,   start, mid,   idx, val);
  else            update(2*node+1, mid+1, end, idx, val);
  tree[node] = tree[2*node] + tree[2*node+1];
}

// Usage:
build(1, 0, n-1);
query(1, 0, n-1, l, r); // sum of arr[l..r]
update(1, 0, n-1, idx, newval);

→ [4,7] → partial → [4,5] → full overlap ✓

          
→ [6,7] → partial → [6,6] → full ✓

Data Structures

        
Lazy Propagation

        
Range updates in O(log n) instead of O(n log n)

        

          Without lazy propagation, updating a range [l,r] requires O((r-l+1)
          log n) time. Lazy propagation defers updates — storing them at nodes
          and only pushing them down when needed.
        


        

          Core idea: Tag a node with a "pending update" and defer it until the
          children are actually needed.
        


        

          
        


        
The Lazy Array

        

          

            Maintain a parallel lazy[] array. When updating node v
            covering [l,r] with value x:
          

          
    
            
  1. Apply the update to tree[v] immediately
    2. 
            
  2. 
              Store the pending update in lazy[v] (don't push to
              children yet)
            
    3. 
            
  3. 
              When we need to visit children of v, first "push down" the lazy
              tag
            
    4. 
          

        


        

          Theory: Lazy propagation trades a constant factor in bookkeeping
          (extra lazy[] storage) for asymptotic savings on range updates.
          Correctness requires that deferred updates be applied in the right
          order; complexity becomes O(log n) per range update or query.
        


        
int tree[4*MAXN], lazy[4*MAXN]; // lazy = pending add value

void pushDown(int node, int start, int end) {
  if (lazy[node] != 0) {
    int mid = (start + end) / 2;
    int lc = 2*node, rc = 2*node+1;
    tree[lc] += lazy[node] * (mid - start + 1); // sum update
    tree[rc] += lazy[node] * (end - mid);
    lazy[lc] += lazy[node];
    lazy[rc] += lazy[node];
    lazy[node] = 0;
  }
}

// Range update: add val to all elements in [l,r]
void rangeUpdate(int node, int start, int end, int l, int r, int val) {
  if (r < start || end < l) return;
  if (l <= start && end <= r) {
    tree[node] += val * (end - start + 1); // apply to this node
    lazy[node] += val;                       // defer to children
    return;
  }
  pushDown(node, start, end); // push before visiting children
  int mid = (start + end) / 2;
  rangeUpdate(2*node,   start, mid,   l, r, val);
  rangeUpdate(2*node+1, mid+1, end, l, r, val);
  tree[node] = tree[2*node] + tree[2*node+1];
}

Data Structures

        
Fenwick Tree (Binary Indexed Tree)

        

          Prefix sums with point updates in O(log n), O(n) space
        

        

          A Fenwick tree is an array-based structure where each index i is
          responsible for storing the sum of a specific range determined by the
          lowest set bit of i.
        


        

          lowbit(i) = i & (-i) → node i covers range [i - lowbit(i) + 1, i]
        


        

          Theory: Fenwick trees exploit binary decomposition of indices using
          lowbit. Both prefix queries and point updates run in O(log n) with
          very low constant factors; building from an array is O(n). Fenwick
          trees are compact and cache-friendly but less general than segment
          trees.
        


        
How Responsibility Works

        

          
        


        
Implementation

        
int bit[MAXN], n;

// Add val to index i (1-indexed)
void update(int i, int val) {
  for (; i <= n; i += i & (-i))
    bit[i] += val;
}

// Sum of [1, i]
int query(int i) {
  int sum = 0;
  for (; i > 0; i -= i & (-i))
    sum += bit[i];
  return sum;
}

// Sum of [l, r]
int rangeSum(int l, int r) {
  return query(r) - query(l - 1);
}

// Build from array in O(n)
void build(vector<int>& arr) {
  for (int i = 1; i <= n; i++) update(i, arr[i-1]);
}

The Invertibility Requirement

          

            Fenwick tree computes query(r) - query(l-1). This
            requires the operation to have an inverse (subtraction for sum, XOR
            for XOR, division for product).
            Min/Max have no inverse, so they can't be answered
            by BIT. Segment tree has no such restriction.
          

        


        

          Theory: Choose a Fenwick tree for minimal code and low constants when
          you only need prefix operations; choose segment trees when you need
          arbitrary range merges, lazy updates, or persistence. Consider space
          and operation invertibility when deciding.
        

      


      
      

        
Advanced

        
Ternary Search on Convex Function

        

          Find the maximum or minimum of a unimodal function in O(log n)
        

        

          Ternary search finds the extremum of a unimodal (strictly convex or
          concave) function. Divide the search space into thirds, eliminate the
          third that cannot contain the answer.
        


        

          Theory: Ternary search requires strict unimodality; otherwise it can
          fail. Each iteration reduces the search interval by ~1/3, giving O(log
          n) iterations; for continuous functions use float comparisons and
          iterate until desired precision.
        


        

          Prerequisite: Function must be strictly unimodal — single peak (for
          max) or valley (for min).
        


        
Visualization

        

          
        


        
Discrete Ternary Search

        
// Maximum of unimodal array
int ternarySearch(vector<int>& arr, int l, int r) {
  while (r - l > 2) {
    int m1 = l + (r - l) / 3;
    int m2 = r - (r - l) / 3;
    if (arr[m1] < arr[m2])
      l = m1;       // peak is to the right of m1
    else
      r = m2;       // peak is to the left of m2
  }
  // Check remaining elements
  int ans = l;
  for (int i = l+1; i <= r; i++)
    if (arr[i] > arr[ans]) ans = i;
  return ans;
}


        
Continuous Ternary Search

        
// Max of continuous unimodal f(x)
double ternarySearch(double l, double r) {
  for (int iter = 0; iter < 200; iter++) { // ~200 suffices
    double m1 = l + (r - l) / 3;
    double m2 = r - (r - l) / 3;
    if (f(m1) < f(m2)) l = m1;
    else r = m2;
  }
  return (l + r) / 2;
}

Advanced

        
Dynamic Programming

        

          Solve complex problems by breaking into overlapping subproblems
        

        

          DP solves problems by combining solutions to overlapping subproblems,
          storing results to avoid recomputation. It's applicable when optimal
          substructure and overlapping subproblems both exist.
        


        

          Theory: DP trades time for space via memoization/tabulation. Identify
          a recurrence and an ordering of states; optimize with space rolling,
          convex hull trick, or divide-and-conquer optimization when applicable.
          Complexity depends on state count × transition cost.
        


        
Two Conditions for DP

        

          

            
Optimal Substructure

            

              The optimal solution to a problem contains optimal solutions to
              its subproblems.
            

            

              Example: In LCS(s,t), if s[i]==t[j], then LCS includes
              LCS(s[0..i-1], t[0..j-1]).
            

          

          

            
Overlapping Subproblems

            

              The same subproblems are solved multiple times in a naive
              recursive approach.
            

            

              Example: Fibonacci — fib(5) needs fib(4) and fib(3), which share
              fib(2) and fib(1).
            

          

        


        
DP Approaches

        

          

            
Top-Down (Memoization)

            

              Recursive + cache. Natural to write, solves only needed states.
            

            
map<int,ll> memo;
ll dp(int i) {
  if (base case) ...
  if (memo.count(i)) return memo[i];
  return memo[i] = ...;
}

          

          

            
Bottom-Up (Tabulation)

            

              Iterative, fill table in dependency order. No recursion overhead.
            

            
dp[0] = base;
for (int i = 1; i <= n; i++)
  dp[i] = f(dp[i-1], dp[i-2], ...);
return dp[n];

// dp[v][0] = max sum in subtree if v not selected
// dp[v][1] = max sum in subtree if v is selected
void dfs(int v, int parent) {
  dp[v][1] = val[v];
  dp[v][0] = 0;
  for (int u : adj[v]) {
    if (u == parent) continue;
    dfs(u, v);
    dp[v][1] += dp[u][0]; // can't take child
    dp[v][0] += max(dp[u][0], dp[u][1]); // best of child
  }
}


        
DP Optimization Techniques

        

          

            
Space Optimization

            

              If dp[i] depends only on dp[i-1], use two variables or rolling
              array to reduce O(n²) space to O(n) or O(1).
            

          

          

            
Divide & Conquer DP

            

              Optimize O(n²) → O(n log n) when opt[i] ≤ opt[i+1] (monotone
              minima).
            

          

          

            
Convex Hull Trick

            

              Optimize DP transitions of form dp[i] = min(dp[j] + b[j]×a[i]) in
              O(n) using convex hull of lines.
